## 1. 이 시스템이 해결하는 문제

아성다이소 매장에서 사고는 **사전에 예측 가능한 조건에서 반복 발생**합니다. 그런데 현재는:
- 사고가 난 뒤 보고 → 뒤늦은 대응
- 전국 1,500+ 매장을 사람이 일일이 모니터링 불가
- 매장마다 위험 프로파일이 다른데 획일적 지침

**SAGO AI는 이것을 바꿉니다**: 오늘의 기상·물동량·과거 사고 패턴을 AI가 분석해 **위험 매장에 내일 사고 전에 맞춤 알림**을 보냅니다.

```
과거 사고 데이터 + 기상 + 물동량
        ↓
   ML 위험점수 산정
        ↓
  임계 초과 매장 탐지
        ↓
 LLM 안전 가이드 생성 (AWS Bedrock)
        ↓
  카카오톡 채널 발송
        ↓
   S3 랜딩 페이지 (수칙 상세)
```

## 2. 지금까지 만들어진 것

### ML 엔진

| 항목 | 현황 |
|------|------|
| 고객 위험점수 모델 | AUC **0.845** (고객 사고 1,481건 학습) |
| 직원 위험점수 모델 | AUC **0.831** (직원 사고 448건 학습) |
| 3축 점수 구조 | S1(조건위험) + S2(사례근접) + S3(심각도) |
| 기상 선제 스캔 | 내일 기상 예보 기반 위험 매장 사전 탐지 |
| θ 자가보정 | ack 피드백으로 임계값 자동 재산정 루프 |

### 알림 시스템

- 위험 임계 초과 매장 → **카카오톡 피드카드 자동 발송**
- AWS Bedrock(Claude)이 매장별 사고 유형에 맞는 **안전수칙 가이드 생성**
- 카드 클릭 → **S3 랜딩 페이지** (수칙 상세 + 캐릭터 애니메이션)

### 대시보드 (관리자용)

- 알림 검토 / 발송 / 모니터링 탭 완성
- KPI 바 + 위험도 필터 + 주간 트렌드 차트
- 수신자 프리셋 관리 + 수신 확인률 + CSV 다운로드
- 카카오맵 매장 위험 지도

## 3. 현재 한계 — 솔직하게

| 한계 | 원인 | 영향 |
|------|------|------|
| **위험점수 S1이 역방향** | 조건 위험 규칙 매핑 오류 | S2(사례근접)가 변별력 전담, S1은 사실상 미작동 |
| **실시간 데이터 없음** | 사내 IT API 미연동 | 과거 패턴 기반만, 실시간 물동량·사고 반영 불가 |
| **ack 데이터 미축적** | 이제 막 발송 시작 | 자가보정·효과 측정 아직 불가 |
| **물동량 추정값 투입** | 실 데이터 미확보 | S3 축 신뢰도 제한적 |

## 4. 풀어야 할 과제

### 🔴 P0 — 지금 당장 결정이 필요한 것

#### ① 사내 IT 본부 API 연동 일정 확정

- 필요 데이터: 실시간 사고 보고, 일별 물동량(매장별)
- 이것 없이는 ML 모델이 추정값 기반으로만 동작
- **결정사항**: IT 본부 담당자 지정 + 데이터 스키마 협의 일정

#### ② 파일럿 발송 매장 선정

- 현재 테스트 단계 — 전국 발송 전에 파일럿 매장 선정 필요
- **결정사항**: 파일럿 매장 수·선정 기준 (고위험 지역? 협조 가능한 점장?)

---

### 🟠 P1 — 이번 고도화 스프린트

#### ③ S1 역변별 문제 해결

- `enrich_leaf_rule`의 위험 레벨이 사고와 반대로 매핑되는 원인 분석·수정
- 해결 시 AUC 추가 상승 기대

#### ④ 산재 승인 DB 탭

- 산재 승인 현황 엑셀 → 대시보드 탭 (1차)
- 이후 HR API 연동 (2차)
- **필요한 것**: 엑셀 파일 or DB 접근 권한

#### ⑤ 알림 효과 측정 체계

- 알림 발송 후 실제 사고 감소 여부 추적
- ack 확인률 + 사후 사고 데이터 매핑
- **결정사항**: 효과 측정 KPI 합의 (무엇을 기준으로 볼 것인가)

---

### 🟡 P2 — 품질 고도화

#### ⑥ 안전수칙 콘텐츠 검증

- 현재 LLM이 생성하는 수칙을 실무자(점장, 안전담당자)가 검토 필요
- 틀린 수칙이 나가면 신뢰 훼손

#### ⑦ 알림 발신업체 추상화

- 현재 카카오 하드코딩 → 문자(SMS) 등 채널 추가 대비

#### ⑧ Higgsfield 영상 연동

- 사고유형별 안전수칙 MP4 영상 → 랜딩 페이지 삽입 (슬롯 이미 구현됨)
- API 키 발급 후 즉시 적용 가능

## 5. GS네오택 역할 정의

### 핵심 미션: ML 모델 정합성 확보

지금 시스템의 **가장 큰 구조적 문제**는 위험점수 3축 중 S1이 역방향으로 작동한다는 것입니다. 위험할수록 점수가 낮게 나오고 있고, S2 혼자 변별력을 감당하고 있습니다. GS네오택이 이 문제를 해결하는 것이 최우선입니다.

### 구체적 과제 (우선순위 순)

#### 1순위 — S1 역변별 근본 해결

- `core/rule_enrichment.py` → 리프 노드별 위험 레벨 매핑 오류 원인 분석
- `processed/incidents_cust.csv`, `incidents_emp.csv` 기반 정합성 재검증
- 해결 시 기대 효과: AUC 0.845 → 0.88+ 상승 + S1·S2·S3 균형 있는 변별력

#### 2순위 — 데이터 품질 검증

- 현재 학습 데이터 1,929건(고객 1,481 + 직원 448)의 라벨링 정확도 검토
- 비사고(negative) 샘플 구성 방식 재검토 (`scripts/build_non_incidents.py`)
- 사고 유형 분류 기준 통일 여부 확인

#### 3순위 — 모델 고도화

- 현재 로지스틱 회귀 기반 → XGBoost / LightGBM 등 앙상블 비교 실험
- Cross-leaf kNN 재정렬 파라미터 최적화
- 물동량 피처 신뢰도 확보 (현재 추정값 → 실 데이터 연동 후 재학습)

### 역할 경계

| 구분 | 내용 |
|------|------|
| **GS네오택 담당** | S1 정합성 분석, 모델 실험, 데이터 품질 검증 |
| **Gerrard팀 유지** | 비즈니스 로직 결정, LLM 가이드 방향, 알림 UX, 배포 |
| **공동** | AUC 평가 기준 합의, 파일럿 발송 결과 해석 |

## 6. GS네오택 온보딩

```
1. GitHub Collaborator 초대
   → https://github.com/gerrardbak08/Project_Sago

2. docs/MANAGER_HANDOFF.md 숙지
   → 아키텍처, 파일 지도, 절대 규칙, Claude 에이전트 활용법

5. 첫 작업 전 실행
   python3 scripts/simulate_triggers.py --source cust --evaluate
   → AUC 0.845 재현 확인 후 S1 분석 시작

6. Claude와 협업 시 첫 마디:
   ".claude/HANDOFF.md 읽어줘" → 현황 즉시 파악
```
