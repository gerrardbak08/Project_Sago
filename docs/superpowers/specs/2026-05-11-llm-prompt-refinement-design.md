# LLM 프롬프트 정교화 — 사례별 인과 분석 설계 문서

**작성일**: 2026-05-11
**상태**: 승인 대기
**관련 스펙**: `2026-05-07-llm-prompt-upgrade-design.md`

---

## 1. 목적 및 배경

### 1.1 배경

현재 시스템은 Decision Tree로 기상·매장 조건별 과거 사고 사례를 검색하고, LLM이 이를 근거로 안전 가이드를 생성한다. 운영해보니 한 가지 근본 문제가 남았다.

**매장 환경 데이터는 고정이고 기상만 매일 바뀐다.** 트리 분기에는 매장 환경 피처가 비중 있게 쓰이므로, 같은 매장은 기상이 웬만큼 바뀌어도 같은 리프(혹은 인접 리프)에 반복 매칭된다. 그 결과 매일 유사한 사례 집합이 LLM에 주입되고, 유사한 가이드가 반복 발송된다.

리프의 각 사고 사례에는 **발생 당시 기상 8개 + 매장 수치 9개 + 매장 범주(형태) + 사고 분류(장소·원인)**가 이미 포함되어 있다. 이 정보를 LLM에 제대로 전달하면, 같은 사례 집합 안에서도 "오늘 조건과 맞는 사례"와 "안 맞는 사례"를 구분해 **오늘 차별화된 가이드**를 생성할 수 있다.

### 1.2 목표

- 각 과거 사례의 **발생 당시 조건**을 유저 프롬프트에 명시한다.
- 기상 피처는 **오늘 대비 Δ**를 미리 계산해 전달한다. 매장 환경은 원본 수치를 그대로 전달한다.
- LLM이 사례별로 **사고 원인 추론 → 오늘 재현 가능성 판단** 절차를 따르도록 시스템 프롬프트를 보강한다.
- 판단 결과를 `오늘의_주의_사례`의 **신규 필드**(`사고_원인_분석`, `오늘_재현_가능성`)에 담아 출력한다.

### 1.3 범위

**In Scope**
- `core/llm.py`
  - `_format_incidents_block`: 사례 포맷을 하이브리드(Δ+매장 원본)로 확장
  - 시스템 프롬프트: "사례 분석 절차" 섹션 추가
  - Tool Spec: `today_alert_cases` 항목에 `cause_analysis`, `today_recurrence_likelihood` 필드 추가
  - Mock 모드: 새 필드 포함해 생성

**Out of Scope**
- 트리 모델·리프 매칭·데이터 파이프라인 변경
- 프론트엔드 수정 (optional 필드로 취급되며 이번 스펙 범위 밖)
- 사고 사례 이미지 생성 (다음 스텝)

## 2. 유저 프롬프트 — 사례 표기 방식

### 2.1 Before

```
- cust_0008 | 계단에서 본인 부주의로 넘어짐
```

리프의 각 사례가 `incident_id | 사고내용` 한 줄로만 나열되어, LLM은 왜 그 사례가 오늘과 관련 있는지 판단할 근거가 부족하다.

### 2.2 After

```
- cust_0008 | 계단에서 본인 부주의로 넘어짐
  [발생일] 2026-01-06
  [발생 당시 기상 (오늘 대비 Δ)]
    temperature_2m_min=-7.8°C (Δ -12.8), temperature_2m_max=2.0°C (Δ -8.0),
    precipitation_sum=0.0mm (Δ 0.0), snowfall_sum=0.0cm (Δ 0.0),
    rain_sum=0.0mm (Δ 0.0), wind_speed_10m_max=1.95m/s (Δ +0.5),
    relative_humidity_2m_mean=58% (Δ -12), soil_temperature_0_to_7cm_mean=-1.0°C (Δ -16.0)
  [발생 당시 매장] 평수=240, 실평수=220, 창고=20, 매장인원=6.8, 입고도우미PO=0.8, 일평균매출=7,206,982, 일평균물동량=144, 형태=직영점
  [분류] 사고유형=낙상, 장소=매장내부, 원인1=부주의
```

- **Δ 계산**: `오늘값 - 발생당시값`. 기상 8개 피처에만 적용.
- **매장 환경**: Δ를 계산하지 않는다. 현재 매장과 과거 사고 매장은 다른 매장일 수 있으므로, 원본 수치를 그대로 보여 LLM이 직접 비교하게 한다.
- **분류 정보**: `사고유형`, `장소`, `원인1`은 소스(cust/emp)에 따라 컬럼명이 다를 수 있으므로, incidents dict에서 존재하는 키만 출력한다.
- **None/결측**: 값이 없으면 `N/A`로 표기하고 Δ는 생략한다.

### 2.3 구현 개요

`core/llm.py`에 헬퍼 함수를 추가한다.

```python
def _format_incident_detail(inc: dict, today_weather: dict) -> str:
    """단일 사례를 다줄 블록으로 포맷. 기상은 Δ 포함, 매장은 원본."""
```

`build_user_prompt`는 이 헬퍼를 호출해 각 사례를 렌더링한다. 기존 `_format_incidents_block`은 제거하거나 이 헬퍼를 사용하도록 교체한다.

## 3. 시스템 프롬프트 — "사례 분석 절차"

기존 "판단 원칙" 섹션 뒤에 다음 블록을 추가한다.

```
## 사례 분석 절차
각 과거 사례에 대해 다음 단계를 수행하십시오.

1. 사고 원인 추론
   - 발생 당시 기상·매장 환경·장소·원인1을 종합해 "이 사고는 왜 일어났는가?"를 판단
   - 원인 유형을 다음 중 하나로 분류:
     (a) 기상 조건 주도  — 결빙·우천·강풍·폭염 등 기상이 직접 원인
     (b) 매장 환경 주도  — 좁은 통로·높은 물동량·인원 부족 등 환경이 직접 원인
     (c) 기상+환경 복합  — 두 요인이 결합해 발생
     (d) 조건 무관 부주의 — 기상·환경과 무관한 일반 부주의

2. 오늘 재현 가능성 판단
   - 기상 Δ가 작을수록 오늘도 유사 조건 → 재현 가능성 높음
   - 기상 Δ가 커도 매장 환경이 오늘 매장과 유사하면 (b) 유형은 여전히 재현 가능
   - 피처 사전의 설명을 참고해, 수치 차이가 실제 안전에 영향을 주는지 판단
   - 판정값: "높음" / "중간" / "낮음"

3. 분류
   - 재현 가능성 높음·원인 유형 (a)(b)(c) → "오늘의 특별 주의사항"의 근거로 사용
   - 재현 가능성 무관·원인 유형 (d) → "상시 주의사항"의 근거로 사용

4. 대표 사례 선정
   - 재현 가능성과 원인 명확성이 높은 사례 중심으로 3~5건을 `오늘의_주의_사례`로 선정
   - 재현 가능성이 낮더라도 대표성 있는 상시 부주의 사례는 포함 가능
   - 향후 incident_id로 사고 이미지가 매칭되므로 ID는 반드시 원본 그대로 사용
```

**임계값**은 주지 않는다. 피처 사전의 설명과 LLM의 추론에 맡긴다.

## 4. 출력 스키마 변경

### 4.1 `오늘의_주의_사례` 항목 확장

**Before**
```json
{
  "incident_id": "cust_0008",
  "사고내용": "계단에서 본인 부주의로 넘어짐",
  "선정_이유": "..."
}
```

**After**
```json
{
  "incident_id": "cust_0008",
  "사고내용": "계단에서 본인 부주의로 넘어짐",
  "사고_원인_분석": "발생 당시 temp_min=-7.8°C·soil_temp=-1.0°C의 결빙 조건에서 계단 낙상. 원인 유형: 기상 조건 주도.",
  "오늘_재현_가능성": "낮음",
  "선정_이유": "오늘은 영상권이라 결빙 재현 가능성은 낮으나, 계단 낙상은 상시 위험이므로 대표 사례로 선정"
}
```

### 4.2 Tool Spec 수정

`SAFETY_GUIDE_TOOL_SPEC`의 `today_alert_cases.items.properties`에 두 필드 추가:

```python
"cause_analysis": {
    "type": "string",
    "description": "사고 원인 분석. 예: '결빙 조건에서 계단 낙상. 원인 유형: 기상 조건 주도.'"
},
"today_recurrence_likelihood": {
    "type": "string",
    "enum": ["높음", "중간", "낮음"],
    "description": "오늘 조건에서 이 사고가 재현될 가능성"
}
```

`required`에 두 필드를 추가한다.

### 4.3 키 매핑 확장

`_KEY_MAP_CASE`에 매핑 추가:

```python
_KEY_MAP_CASE = {
    "incident_id": "incident_id",
    "incident_content": "사고내용",
    "cause_analysis": "사고_원인_분석",          # 신규
    "today_recurrence_likelihood": "오늘_재현_가능성",  # 신규
    "selection_reason": "선정_이유",
}
```

`_translate_response`의 `today_alert_cases` 변환은 현재 로직 그대로 동작한다 (모든 키를 `_KEY_MAP_CASE`로 매핑).

## 5. Mock 모드 업데이트

`generate_guide_mock`에서 `오늘의_주의_사례` 항목에 새 필드를 포함한다.

```python
picks.append({
    "incident_id": iid,
    "사고내용": content,
    "사고_원인_분석": "(Mock) 유사 조건에서 발생한 대표 사례",
    "오늘_재현_가능성": "중간",
    "선정_이유": "유사 조건에서 발생한 대표 사고 사례."
})
```

재현 가능성은 고정값("중간")으로 둔다. Mock은 규칙 기반이므로 정교한 판단을 하지 않는다.

## 6. 데이터 흐름 변경

**Before**
```
build_user_prompt
  → _format_incidents_block  (incident_id | 사고내용)
  → LLM (리프 요약 + 한 줄 사례 리스트)
  → today_alert_cases [{incident_id, 사고내용, 선정_이유}]
```

**After**
```
build_user_prompt
  → _format_incident_detail  (사례별 Δ 계산 + 매장 원본 + 분류)
  → LLM (리프 요약 + 풍부한 사례 블록 + 사례 분석 절차 지시)
  → today_alert_cases [{incident_id, 사고내용, 사고_원인_분석, 오늘_재현_가능성, 선정_이유}]
```

## 8. 테스트 전략

- **Mock 단위 검증**: `USE_MOCK_LLM=true` 환경에서 가이드 생성 → 출력 JSON에 `사고_원인_분석`, `오늘_재현_가능성` 필드가 모든 `오늘의_주의_사례` 항목에 포함되는지 확인.
- **유저 프롬프트 포맷 검증**: `build_user_prompt`를 단독 호출해 한 사례 이상의 Δ 계산과 매장 원본이 올바르게 출력되는지 육안 확인.
- **Bedrock 실호출**: 매장 1건으로 3가지 날짜(맑음·우천·혹한)로 실행해 사례별 `오늘_재현_가능성`이 날짜에 따라 달라지는지 확인.
- **동일 매장·연속 3일**: 연속 3일치 가이드가 차별화되는지 (동일 리프 매칭이어도 LLM 출력이 다른지) 육안 비교.

## 9. 다음 스텝 (본 스펙 범위 밖)

- 대시보드 사례 카드에 이미지 + `사고_원인_분석` + `오늘_재현_가능성` 배지 노출
- 운영 데이터 축적 후 재현 가능성 판단 정확도 검토
