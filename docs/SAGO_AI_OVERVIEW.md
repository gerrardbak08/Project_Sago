# SAGO AI 서비스 소개서

## 1. 서비스 개요

SAGO AI는 아성다이소 매장의 고객 사고와 직원 산업재해 데이터를 기반으로, 매장별 안전 리스크를 분석하고 당일 조건에 맞는 안전 가이드를 생성해 발송하는 AI 안전보건 서비스입니다.

기존 안전관리가 사고 발생 후 집계와 보고에 머무르기 쉬웠다면, SAGO AI는 과거 사고 사례, 매장 특성, 기상 조건을 함께 분석해 "오늘 이 매장에서 특히 조심해야 할 사고"를 사전에 안내합니다. 이를 통해 매장 운영자는 사고 예방 행동을 더 구체적으로 실행할 수 있고, 안전보건 담당자는 발송 현황과 매장별 위험 유형을 대시보드에서 확인할 수 있습니다.

현재 구현된 서비스는 안전보건 통합 대시보드, 사고 사례 기반 AI 안전 가이드 생성, 카카오 메시지 발송, 발송 이력 모니터링까지 포함하는 PoC 형태입니다.

## 2. 해결하려는 문제

매장 안전사고는 단순히 하나의 원인으로 발생하지 않습니다. 비나 눈 같은 기상 조건, 매장 면적, 창고 규모, 물동량, 근무 인원, 고객 혼잡도, 과거 사고 패턴이 함께 작용합니다.

하지만 일반적인 안전 공지는 모든 매장에 동일한 문구를 보내는 경우가 많아, 현장에서 체감하는 실효성이 떨어질 수 있습니다. 또한 사고 데이터가 쌓여 있어도 이를 매일의 운영 의사결정으로 연결하기 어렵습니다.

SAGO AI는 이 문제를 다음 방식으로 해결합니다.

- 과거 고객 사고와 직원 사고를 각각 분석해 사고 유형별 패턴을 분리합니다.
- 매장별 고정 정보와 당일 기상 정보를 결합해 유사한 과거 사고 사례를 찾습니다.
- LLM이 유사 사례의 원인과 오늘의 재현 가능성을 분석해 구체적인 안전 수칙을 작성합니다.
- 생성된 가이드를 카카오 메시지 형태로 발송하고, 발송 결과를 대시보드에 기록합니다.

## 3. 핵심 사용자

SAGO AI는 안전보건팀, 본부 관리자, 영업 조직, 매장 운영 담당자를 주요 사용자로 둡니다.

안전보건팀은 전체 사고 현황과 발송 이력을 관리하고, 매장별로 어떤 위험 유형이 반복되는지 확인할 수 있습니다. 본부와 영업 조직은 부서, 팀, 지역, 매장 단위로 사고 현황을 파악하고 우선 관리 대상을 찾을 수 있습니다. 매장 담당자는 당일 위험 사례와 예방 수칙을 간단한 메시지 형태로 받아 현장 조치에 활용할 수 있습니다.

## 4. 주요 기능

### 4.1 안전보건 통합 대시보드

React, Vite, Tailwind CSS 기반의 웹 대시보드입니다. 근로자 사고와 고객 사고를 분리해 분석하며, 역할별로 필요한 화면에 접근할 수 있도록 구성되어 있습니다.

근로자 사고 영역은 요약, 부서·팀, 매장 IR, 파트장, 매장위험지도, 시계열, 요인 분석, 인적요인, 재발재해자, 의료 심각도, 법적 보고, 비용 손실 등의 탭으로 구성됩니다.

고객 사고 영역은 요약, 부서별 분석, 유형·장소 분석, 보상, 모니터링, 피해자 분석 탭을 제공합니다.

### 4.2 사고 사례 기반 AI 안전 가이드

서비스는 고객 사고(CUST)와 직원 사고(EMP)를 별도 모델로 처리합니다. 현재 모델 산출물 기준으로 CUST는 1,481건, EMP는 448건의 사고 사례를 사용하며, 각 사고에는 발생일, 사고 내용, 사고 유형, 장소, 원인, 매장 정보, 기상 정보, 이미지 매칭 정보가 포함됩니다.

Decision Tree는 사고를 직접 예측하기 위한 모델이 아니라, 유사 조건의 과거 사고 사례를 찾기 위한 조건부 사례 검색 모델로 사용됩니다. 즉 "사고가 날 확률"을 숫자로 예측하기보다, "이 매장과 오늘 조건에서 참고할 만한 과거 사고 묶음"을 찾아 LLM에 전달합니다.

LLM은 리프 노드의 사고 사례와 규칙 기반 위험 해석을 바탕으로 다음 내용을 생성합니다.

- 위험 요약
- 주요 위험 유형
- 오늘의 주의사항 3~5건
- 사고 원인 분석
- 오늘 재현 가능성
- 구체적인 안전 수칙
- 관련 피처
- 부주의 사례를 종합한 상시 안전 수칙

### 4.3 사고 이미지 매칭

사고 사례에는 `incident_id`가 부여되어 있으며, 이미지 파일명과 사고 내용을 TF-IDF 코사인 유사도로 비교해 `image_url`을 매칭합니다. 현재 이미지 자산은 190개가 준비되어 있고, LLM이 선정한 오늘의 주의 사례에는 후처리로 이미지 URL이 붙습니다.

이 구조 덕분에 알림은 단순 텍스트 안내를 넘어, 유사 사고 상황을 시각적으로 보여주는 형태로 확장됩니다.

### 4.4 카카오 안전 가이드 발송

관리자는 알림 발송 화면에서 영업 중인 매장을 검색해 여러 개 선택하고, 날짜와 카카오 친구 UUID를 입력해 안전 가이드를 발송할 수 있습니다.

발송 과정은 다음 순서로 동작합니다.

1. 선택한 매장과 날짜를 notify Lambda에 전달합니다.
2. Lambda가 매장 정보와 모델 파일을 S3에서 로드합니다.
3. 매장 위치와 날짜를 기준으로 기상 정보를 조회합니다.
4. 고객 사고 모델과 직원 사고 모델을 각각 매칭합니다.
5. Bedrock 기반 LLM 또는 Mock 모드로 안전 가이드를 생성합니다.
6. 대표 사고 사례와 이미지를 포함한 카카오 메시지 템플릿을 구성합니다.
7. 카카오 친구 메시지 API로 발송합니다.
8. 발송 성공/실패 결과와 상세 가이드를 S3에 저장합니다.

### 4.5 알림 현황 모니터링

알림 현황 화면은 날짜별 발송 기록을 조회하고, 매장별 발송 상태, 채널, 주요 위험 유형, 상세 가이드를 확인합니다.

상세 화면에서는 고객 안전 알림과 직원 안전 알림을 카카오톡 채팅창 형태로 미리 볼 수 있습니다. 각 메시지는 유사 사고 사례, 사고 이미지, 안전 조치 문구를 포함하며, 여러 사례가 있을 경우 캐러셀 방식으로 확인할 수 있습니다.

## 5. 데이터 및 모델 구조

현재 서비스는 원본 엑셀 데이터에서 매장 정보, 고객 사고, 직원 사고, 인원 현황, 매출, 물동량을 전처리해 CSV와 JSON 산출물로 변환합니다.

핵심 피처는 총 18개입니다.

- 기상 피처 8개: 최저기온, 최고기온, 강수량, 적설량, 강우량, 최대풍속, 평균습도, 지면온도
- 매장 수치 피처 9개: 평수, 실평수, 진열평수, 창고, 계약면적, 매장인원, 입고도우미 인원, 일평균매출, 일평균물동량
- 매장 범주 피처 1개: 매장 형태

모델은 CUST와 EMP를 분리해 학습합니다.

- CUST 모델: 고객 사고 사례 기반, 현재 56개 리프
- EMP 모델: 직원 사고 사례 기반, 현재 23개 리프
- 런타임에서는 scikit-learn 없이 JSON으로 export된 트리 규칙을 순수 Python으로 실행합니다.
- 직접 리프 매칭에 실패하면 sibling 리프 병합, 전체 리프 병합 순서로 fallback합니다.

## 6. AI 가이드 생성 방식

SAGO AI의 LLM 프롬프트는 단순히 사고 목록을 전달하는 방식이 아니라, 사고별 발생 당시 조건과 오늘 조건의 차이를 함께 제공합니다.

각 사고 사례에는 다음 정보가 포함됩니다.

- 사고 내용과 사고 유형
- 발생 당시 기상 정보
- 오늘 기상 대비 차이
- 발생 당시 매장 환경
- 장소, 원인, 기인물 등 분류 정보
- 이미지 매칭 URL

또한 `core/rule_enrichment.py`는 트리 리프의 수치 조건을 도메인 지식으로 해석합니다. 예를 들어 강수량, 적설량, 풍속, 창고 규모, 물동량, 매장인원 등의 조건을 안전 리스크 문장으로 변환해 LLM에 전달합니다.

이로써 LLM은 단순한 일반 안전 문구가 아니라, "왜 이 사례가 오늘 다시 발생할 수 있는지"를 설명하고, 현장에서 바로 실행할 수 있는 수칙을 생성합니다.

## 7. 시스템 아키텍처

현재 구현은 정적 웹 프론트엔드와 서버리스 백엔드로 구성되어 있습니다.

- 프론트엔드: React, Vite, Tailwind CSS, Recharts
- 데이터 처리: Python, pandas, scikit-learn
- 모델 산출물: JSON 기반 Decision Tree 규칙, leaf table, metadata, encoder map, siblings
- AI 생성: AWS Bedrock 기반 LLM, Mock fallback 지원
- 기상 데이터: Open-Meteo API
- 백엔드: AWS Lambda Function URL
- 저장소: S3 모델 버킷, daily/alerts 결과 버킷, 정적 웹 호스팅 버킷
- 배치: EventBridge 기반 일 단위 batch-orchestrator Lambda
- 알림 채널: 카카오 친구 메시지 API
- 배포: Terraform + `deploy.sh`

구현된 Lambda는 크게 세 가지입니다.

- `notify`: 매장별 안전 가이드 생성, 카카오 발송, 발송 이력 저장
- `alerts`: 날짜별 알림 목록과 상세 파일 조회
- `batch`: 매일 아침 영업 매장을 순회하며 안전 가이드 생성 및 기록

## 8. 서비스 흐름

전체 흐름은 다음과 같습니다.

1. 원본 사고·매장·인원·매출 데이터를 전처리합니다.
2. 사고별 기상 정보를 결합하고, 고객 사고와 직원 사고 데이터셋을 구성합니다.
3. Decision Tree를 학습해 유사 조건별 사고 사례 그룹을 만듭니다.
4. 모델 산출물과 매장 정보를 S3에 업로드합니다.
5. 관리자가 대시보드에서 매장과 날짜를 선택합니다.
6. Lambda가 당일 기상과 매장 특성을 기반으로 유사 사고 리프를 찾습니다.
7. LLM이 오늘의 위험 유형, 유사 사고, 재현 가능성, 안전 수칙을 생성합니다.
8. 카카오 메시지로 발송하고, 결과를 S3에 저장합니다.
9. 대시보드에서 발송 현황과 상세 가이드를 조회합니다.

## 9. 기대 효과

SAGO AI는 안전관리를 사후 보고 중심에서 사전 예방 중심으로 전환합니다.

첫째, 매장별 상황에 맞는 안전 가이드를 제공합니다. 모든 매장에 같은 문구를 보내는 것이 아니라, 매장 규모, 물동량, 근무 인원, 기상 조건, 과거 사고 사례를 반영합니다.

둘째, 사고 데이터를 현장 행동으로 연결합니다. 축적된 사고 DB가 단순 통계가 아니라, 오늘의 안전 조치 문구와 유사 사고 이미지로 재활용됩니다.

셋째, 안전보건 담당자의 운영 효율을 높입니다. 발송 대상 선택, 가이드 생성, 카카오 발송, 이력 조회가 하나의 대시보드 흐름으로 연결됩니다.

넷째, 고객 사고와 직원 사고를 함께 관리합니다. 고객 안전과 근로자 안전을 분리 분석하면서도, 하나의 서비스 안에서 통합적으로 모니터링할 수 있습니다.

## 10. 현재 구현 범위

현재 코드 기준 구현된 범위는 다음과 같습니다.

- React 기반 안전보건 통합 대시보드
- 근로자 사고 분석 탭과 고객 사고 분석 탭
- 관리자용 알림 현황/알림 발송 탭
- 매장 검색 및 다중 매장 선택
- 카카오 친구 UUID 기반 테스트 발송
- 고객 사고/직원 사고 별도 Decision Tree 모델
- JSON 트리 규칙 기반 Lambda 런타임 매칭
- Bedrock LLM 안전 가이드 생성 및 Mock fallback
- 사고 사례별 원인 분석, 재현 가능성, 안전 수칙 생성
- 사고 사례 이미지 매칭 및 카카오 미리보기
- S3 기반 알림 상세/인덱스 저장
- alerts Lambda를 통한 발송 현황 조회
- Terraform 인프라 정의 및 배포 스크립트

현재 PoC는 실제 운영 전 단계의 검증용 구현입니다. 특히 카카오 발송은 친구 UUID 기반 테스트 흐름이며, 정식 운영을 위해서는 카카오 비즈니스 채널, 수신자 DB, 권한 관리, 개인정보 처리 정책, 발송 승인 플로우가 추가로 필요합니다.

## 11. 한 줄 소개

SAGO AI는 다이소 매장의 과거 사고 데이터와 당일 환경 조건을 연결해, 매장별로 오늘 필요한 안전 가이드를 생성하고 카카오 알림으로 전달하는 AI 기반 안전사고 예방 서비스입니다.


# SAGO AI 상세 로직 설명서

## 1. 전체 처리 흐름

SAGO AI의 핵심 로직은 "과거 사고 데이터를 조건별 사례 그룹으로 만들고, 오늘의 매장·기상 조건에 가장 가까운 사고 사례를 찾아 AI 안전 가이드로 변환하는 것"입니다.

전체 흐름은 다음과 같습니다.

1. 원본 엑셀 데이터를 전처리해 매장, 고객 사고, 직원 사고 데이터를 정리합니다.
2. 사고 발생일의 기상 데이터를 결합합니다.
3. 고객 사고(CUST), 직원 사고(EMP)를 분리해 Decision Tree를 학습합니다.
4. 트리를 JSON 규칙으로 export하고, 각 리프에 사고 사례 목록을 저장합니다.
5. 알림 요청이 들어오면 매장 정보와 당일 기상 정보를 조합합니다.
6. JSON 트리 규칙을 순수 Python으로 실행해 리프를 찾습니다.
7. 리프 규칙을 도메인 지식으로 해석해 위험 시나리오를 만듭니다.
8. 리프에 포함된 사고 사례와 오늘 조건을 LLM에 전달합니다.
9. LLM이 오늘의 주의사항, 부주의 주의사항, 재현 가능성, 안전 수칙을 생성합니다.
10. 사고 사례 이미지 URL을 후처리로 붙입니다.
11. 카카오 메시지로 발송하고, 발송 결과를 S3에 기록합니다.

## 2. 데이터 전처리 로직

전처리는 `scripts/build_dataset.py`가 담당합니다. 산출물은 `processed/` 아래 CSV 파일입니다.

### 2.1 매장 마스터 구축

`매장리스트_260408.xlsx`를 기준으로 매장 마스터를 만들고, 인원 현황과 매출·물동량 데이터를 결합합니다.

주요 처리 내용은 다음과 같습니다.

- 매장코드 기준으로 매장 기본 정보를 로드합니다.
- 기존 `processed/stores.csv`가 있으면 이미 확보된 위도·경도를 재사용합니다.
- `KAKAO_REST_API_KEY`가 있으면 카카오 주소 검색 API로 위도·경도를 보강합니다.
- 직영점/유통점 인원 현황에서 `합산PO`, `입고도우미PO`를 추출합니다.
- `합산PO`는 `매장인원`으로 사용합니다.
- 직영점 일매출 데이터에서 `일평균매출`, `일평균물동량`을 계산합니다.

이 결과로 매장별 위치, 규모, 인력, 매출, 물동량 정보가 하나의 테이블로 만들어집니다.

### 2.2 사고 데이터 정제

고객 사고와 직원 사고는 별도 파일에서 로드한 뒤 매장 정보와 매칭합니다.

고객 사고(CUST)는 다음 정보를 사례 컬럼으로 사용합니다.

- 매장, 매장명, 지역
- 발생일시
- 사고유형
- 장소
- 원인1, 원인2, 원인3
- 사고내용요약

직원 사고(EMP)는 다음 정보를 사례 컬럼으로 사용합니다.

- 매장, 매장명, 지역
- 발생일시
- 재해 종류
- 재해 유형
- 기인물
- 상병명
- 사고 내용

고객 사고의 일부 오타는 정규화합니다. 예를 들어 `중돌`은 `충돌`, `낙성`과 `추락`은 `낙상`으로 보정합니다.

### 2.3 기상 데이터 결합

사고 발생일과 매장 위치를 기준으로 Open-Meteo 기상 데이터를 수집합니다.

모델에 실제로 사용하는 핵심 기상 피처는 8개입니다.

- `temperature_2m_min`
- `temperature_2m_max`
- `precipitation_sum`
- `snowfall_sum`
- `rain_sum`
- `wind_speed_10m_max`
- `relative_humidity_2m_mean`
- `soil_temperature_0_to_7cm_mean`

전처리 결과 각 사고 사례는 "사고 내용 + 발생 당시 매장 정보 + 발생 당시 기상 정보"를 함께 가진 레코드가 됩니다.

### 2.4 사고 ID와 이미지 URL

각 사고에는 `incident_id`가 붙습니다. 고객 사고는 `cust_0001`, 직원 사고는 `emp_0001` 형식입니다.

`scripts/match_images.py`는 사고 내용과 이미지 파일명의 키워드를 TF-IDF 코사인 유사도로 비교해 `image_url` 컬럼을 채웁니다. 이 값은 나중에 LLM이 선정한 사례에 이미지 URL을 붙이는 데 사용됩니다.

## 3. 모델 학습 로직

모델 학습은 `scripts/train.py`가 담당합니다.

SAGO AI에서 Decision Tree는 일반적인 의미의 예측 모델이라기보다, 조건별 사고 사례를 찾기 위한 검색 인덱스 역할을 합니다.

즉 목표는 "오늘 사고가 날 확률"을 직접 계산하는 것이 아니라, "오늘 조건과 유사한 과거 사고 묶음"을 찾는 것입니다.

### 3.1 입력 피처

트리 학습에는 총 18개 피처를 사용합니다.

기상 피처 8개:

- 최저기온
- 최고기온
- 강수량
- 적설량
- 강우량
- 최대풍속
- 평균습도
- 지면온도

매장 수치 피처 9개:

- 평수
- 실평수
- 진열평수
- 창고
- 계약면적
- 매장인원
- 입고도우미PO
- 일평균매출
- 일평균물동량

매장 범주 피처 1개:

- 형태

`형태`는 `유통점`, `유통행사`, `직영점` 순서로 숫자 인코딩합니다.

### 3.2 라벨

라벨은 사고 유형 분포가 잘 나뉘는 조건을 찾기 위한 기준으로만 사용합니다.

- CUST 라벨: `사고유형`
- EMP 라벨: `재해 유형`

이 라벨을 직접 예측해 운영 화면에 노출하는 구조가 아닙니다. 라벨은 트리가 어떤 조건에서 사고 유형 분포가 달라지는지 분기하도록 돕는 역할입니다.

### 3.3 하이퍼파라미터 튜닝

현재 학습 스크립트는 후보 파라미터 조합을 만들어 train/test split으로 평가합니다.

후보 범위는 다음과 같습니다.

- `criterion`: `gini`, `entropy`
- `max_depth`: 15, 20, 25
- `min_samples_leaf`: 15, 20
- `class_weight`: `balanced`
- `random_state`: 42

정렬 기준은 macro F1, accuracy, 리프 수의 적정성, 트리 깊이입니다.

현재 모델 메타데이터 기준 결과는 다음과 같습니다.

- CUST: 1,481건, 56개 리프, 최소 리프 크기 20건
- EMP: 448건, 23개 리프, 최소 리프 크기 15건

### 3.4 모델 산출물

학습 결과는 `models/{cust,emp}/` 아래 JSON으로 저장됩니다.

- `tree_rules.json`: scikit-learn 없이 실행 가능한 트리 분기 구조
- `leaf_table.json`: 리프별 규칙, 사고 통계, 사고 사례 목록
- `metadata.json`: 피처명, 사고 건수, 리프 통계, 하이퍼파라미터
- `encoder_map.json`: 범주형 피처 인코딩 정보
- `siblings.json`: fallback 시 사용할 내부 노드별 하위 리프 목록

Lambda 런타임은 scikit-learn 모델 파일을 로드하지 않고, 이 JSON 산출물만 사용합니다.

## 4. 리프 매칭 로직

리프 매칭은 `core/rule_matcher.py`가 담당합니다.

### 4.1 피처 구성

알림 요청이 들어오면 Lambda는 매장 정보와 기상 정보를 합쳐 트리에 넣을 feature dict를 만듭니다.

기상 피처는 `core.weather.get_weather()` 결과에서 가져오고, 매장 수치 피처는 `stores.json`에서 가져옵니다. 결측값은 기본적으로 0 또는 인코더 기본값으로 보정합니다.

매장 형태는 `encoder_map.json`의 `형태` 매핑을 사용합니다. 알 수 없는 값은 기본적으로 `직영점` 코드에 가깝게 처리합니다.

### 4.2 JSON 트리 실행

`tree_rules.json`에는 각 노드가 다음 형태로 저장되어 있습니다.

- split 노드: feature, threshold, left, right
- leaf 노드: leaf_id, samples

런타임은 root 노드에서 시작해 다음 규칙을 반복합니다.

```text
feature 값 <= threshold 이면 left 노드로 이동
feature 값 > threshold 이면 right 노드로 이동
leaf 노드에 도달하면 leaf_id 반환
```

이 방식 덕분에 Lambda에는 scikit-learn 의존성이 필요 없습니다.

### 4.3 Fallback

계산된 leaf_id가 `leaf_table.json`에 있으면 Level 0으로 바로 사용합니다.

만약 직접 리프 조회에 실패하면 다음 순서로 fallback합니다.

1. Level 1: `siblings.json`에서 해당 리프를 포함하는 가장 작은 형제 리프 그룹을 찾아 병합합니다.
2. Level 2: 전체 리프를 병합해 글로벌 사례 풀을 만듭니다.

병합 시에는 사고 사례 리스트와 summary 분포를 합산합니다.

## 5. 리프 위험 해석 로직

리프 매칭 후에는 `core/rule_enrichment.py`가 트리 규칙을 도메인 언어로 해석합니다.

트리 규칙은 원래 다음처럼 수치 조건입니다.

```text
temperature_2m_min <= 0.0 & precipitation_sum > 3.0 & 일평균물동량 > 325
```

이 규칙을 그대로 LLM에 주면 의미가 약하기 때문에, 서비스는 각 조건을 안전 리스크 표현으로 변환합니다.

예시는 다음과 같습니다.

- `temperature_2m_min <= 0`: 한파, 결빙 위험, 입구·주차장 낙상 증가
- `precipitation_sum > 10`: 많은비, 바닥 침수와 미끄러짐 위험
- `wind_speed_10m_max > 10`: 폭풍, 외부 작업 중단 권고
- `창고 > 35`: 초대규모 창고, 복잡 동선과 중장비 사고 위험
- `일평균물동량 > 325`: 고물동량, 입고·진열 강도 증가
- `매장인원 <= 5`: 인원 부족, 1인 작업과 과로 위험

해석 결과는 다음 필드로 구성됩니다.

- `scenario_label`: 종합 위험 시나리오
- `risk_level`: 높음, 중간, 낮음
- `primary_risk_factors`: 핵심 위험요인
- `weather_context`: 기상 조건 요약
- `store_context`: 매장 환경 요약
- `dominant_accident_type`: 과거 최다 사고유형
- `explanation`: LLM 프롬프트에 넣을 종합 설명

위험등급은 심각도 높은 조건의 개수로 결정합니다.

- 핵심 위험요인 2개 이상: 높음
- 핵심 위험요인 1개: 중간
- 핵심 위험요인 없음: 낮음

## 6. LLM 프롬프트 구성 로직

LLM 로직은 `core/llm.py`가 담당합니다.

### 6.1 시스템 프롬프트

시스템 프롬프트는 대형 유통매장 안전관리 전문가 역할을 부여합니다.

핵심 지시는 다음과 같습니다.

- 과거 사고를 오늘 조건과 연관된 사례와 조건 무관 부주의 사례로 구분합니다.
- 기상뿐 아니라 매장 환경 피처도 반드시 분석합니다.
- 리프 노드 위험 분석을 참고해 사고 재현 가능성을 판단합니다.
- 오늘 조건과 연관된 사례는 `today_precautions`로 출력합니다.
- 부주의 사례는 `negligence_precautions`로 종합 안전 수칙을 작성합니다.
- 출력은 Bedrock Tool Use JSON 스키마를 따릅니다.

### 6.2 유저 프롬프트

유저 프롬프트는 요청마다 동적으로 생성됩니다.

포함되는 정보는 다음과 같습니다.

1. 오늘의 매장 조건
2. 오늘의 기상 조건
3. 리프 노드 위험 분석
4. 유사 조건 과거 사고 사례의 분포
5. 각 사고 사례의 상세 정보
6. 출력 지시

특히 사고 사례는 단순히 한 줄 텍스트로 전달하지 않습니다. 각 사례마다 발생 당시 기상과 오늘 기상의 차이를 함께 전달합니다.

예시 구조는 다음과 같습니다.

```text
- cust_0008 | 계단에서 넘어짐
  [발생일] 2025-01-06
  [발생 당시 기상 (오늘 대비 Δ)] temperature_2m_min=-7.8°C (Δ +5.0), ...
  [발생 당시 매장] 평수=240, 매장인원=6.8, 일평균물동량=144, ...
  [분류] 사고유형=낙상, 장소=매장내부, 원인1=부주의
```

Δ는 `오늘값 - 발생당시값`입니다. Δ가 0에 가까울수록 오늘과 유사한 조건으로 볼 수 있습니다.

### 6.3 Tool Use 출력 스키마

Bedrock 호출은 Converse API와 Tool Use를 사용합니다. LLM은 자유 텍스트가 아니라 정해진 JSON 스키마로 응답해야 합니다.

상위 필드는 다음과 같습니다.

- `risk_summary`
- `main_risk_type`
- `today_precautions`
- `negligence_precautions`
- `additional_notes`

응답을 받은 뒤 `_translate_response()`가 영어 키를 한글 키로 변환합니다.

최종 guide 구조는 다음과 같습니다.

- `위험_요약`
- `주요_위험유형`
- `오늘의_주의사항`
- `부주의_주의사항`
- `추가_참고`

`오늘의_주의사항`의 각 항목은 다음 정보를 포함합니다.

- `incident_id`
- `사고내용`
- `사고_원인_분석`
- `오늘_재현_가능성`
- `수칙`
- `관련_피처`
- `image_url`

### 6.4 Mock fallback

`USE_MOCK_LLM=true`이거나 Bedrock 호출에 실패하면 Mock 모드로 전환됩니다.

Mock 모드는 다음 기상 규칙을 기준으로 기본 안전 가이드를 만듭니다.

- 최저기온이 0도 미만이면 결빙 낙상 수칙
- 강수량이 0보다 크면 우천 미끄러짐 수칙
- 적설량이 0보다 크면 낙설·미끄러짐 수칙
- 최대풍속이 10m/s보다 크면 강풍 전도 수칙

Mock 결과도 실제 LLM 결과와 동일한 한글 스키마를 유지합니다.

### 6.5 이미지 URL 후처리

LLM이 guide를 생성한 뒤 `_attach_image_urls()`가 실행됩니다.

이 함수는 리프의 사고 사례 목록에서 `incident_id`별 `image_url` 맵을 만들고, LLM이 선정한 `오늘의_주의사항` 항목에 같은 `incident_id`의 이미지 URL을 붙입니다.

즉 LLM은 어떤 사고 사례를 대표로 고를지만 결정하고, 실제 이미지 URL 연결은 코드가 결정적으로 처리합니다.

## 7. notify Lambda 로직

수동 발송은 `lambdas/notify/handler.py`가 담당합니다.

### 7.1 입력

프론트엔드는 다음 형식으로 요청합니다.

```json
{
  "store_codes": [10130, 10481],
  "date": "2026-05-06",
  "channel": "kakao",
  "receiver_uuids": ["..."]
}
```

`channel`이 `kakao`이면 `receiver_uuids`가 필수입니다.

### 7.2 매장별 처리

Lambda는 요청된 매장코드를 순회하며 다음 작업을 수행합니다.

1. `stores.json`에서 매장 정보를 찾습니다.
2. 위도·경도가 없으면 해당 매장은 실패 처리합니다.
3. `get_weather()`로 날짜 기준 기상 정보를 가져옵니다.
4. CUST와 EMP 각각에 대해 모델 파일을 로드합니다.
5. feature dict를 만들고 `match_with_fallback()`으로 리프를 찾습니다.
6. `generate_guide()`로 안전 가이드를 생성합니다.
7. 카카오 템플릿을 구성합니다.
8. 카카오 친구 메시지 API로 발송합니다.
9. 상세 결과와 인덱스를 S3에 저장합니다.

### 7.3 카카오 메시지 구성

카카오 메시지는 feed 템플릿으로 구성됩니다.

대표 사례 선택은 직원 안전(EMP)을 먼저 보고, 없으면 고객 안전(CUST)을 사용합니다.

메시지에는 다음이 포함됩니다.

- 매장명
- 날짜
- 대표 사고 내용
- 대표 수칙
- 대표 이미지
- 대시보드 상세 링크

이미지가 로컬 경로이면 `FRONTEND_URL`을 붙여 공개 URL로 변환한 뒤 카카오 이미지 업로드 API에 업로드합니다. 이미지가 없으면 fallback 이미지 URL을 사용합니다.

### 7.4 발송 이력 저장

발송 후 `_record_alert()`가 S3에 결과를 저장합니다.

상세 파일:

```text
alerts/{date}/{store_code}_{timestamp}.json
```

인덱스 파일:

```text
alerts/{date}/index.json
```

인덱스에는 다음 요약 정보가 저장됩니다.

- 매장코드
- 매장명
- 지역
- 날짜
- 발송 시각
- trigger_type
- channel
- 수신자
- 성공/실패 수신자
- delivery_status
- CUST 주요 위험유형
- EMP 주요 위험유형
- 상세 파일 경로

## 8. alerts Lambda 로직

`lambdas/alerts/handler.py`는 알림 현황 조회를 담당합니다.

Lambda Function URL 경로에 따라 S3 파일을 읽어 반환합니다.

- `GET /{date}`: `alerts/{date}/index.json` 반환
- `GET /{date}/{filename}`: `alerts/{date}/{filename}` 반환

프론트엔드의 알림 현황 화면은 이 API를 사용해 날짜별 발송 목록과 상세 가이드를 조회합니다.

## 9. batch Lambda 로직

`lambdas/batch/handler.py`는 EventBridge로 실행되는 일 단위 배치 로직입니다.

현재 구조는 notify와 거의 같은 가이드 생성 흐름을 사용합니다.

차이는 다음과 같습니다.

- 실행 날짜는 Lambda 실행 시점의 KST 날짜를 사용합니다.
- `BATCH_STORE_CODES` 환경변수가 있으면 지정 매장만 처리합니다.
- 없으면 전체 영업 매장을 대상으로 합니다.
- 채널은 `NOTIFY_CHANNEL` 환경변수를 사용합니다.
- 전체 결과는 `daily/{date}/results.json`에도 저장합니다.

현재 batch의 수신자 목록은 비어 있으며, 실제 운영 시 직원 연락처 DB 연동이 필요한 상태입니다.

## 11. 배포 로직

배포는 `deploy.sh`와 Terraform이 담당합니다.

`deploy.sh`의 주요 단계는 다음과 같습니다.

1. AWS 자격증명을 갱신합니다.
2. Lambda Layer용 `core-layer.zip`을 만듭니다.
3. `batch.zip`, `notify.zip`, `alerts.zip`을 만듭니다.
4. Terraform init/apply를 실행합니다.
5. Terraform output에서 notify URL, alerts URL, frontend URL을 읽습니다.
6. `proj/.env.production`에 Vite 환경변수를 반영합니다.
7. 모델 산출물을 S3 모델 버킷에 업로드합니다.
8. React 앱을 빌드하고 S3 정적 웹 호스팅 버킷에 업로드합니다.
9. 사고 사례 이미지를 S3 `images/` 경로에 업로드합니다.
10. `stores.json`을 모델 버킷에 업로드합니다.

Terraform은 다음 리소스를 정의합니다.

- 프론트엔드 S3 버킷
- 모델 S3 버킷
- daily/alerts 결과 S3 버킷
- notify Lambda Function URL
- alerts Lambda Function URL
- batch-orchestrator Lambda
- EventBridge daily batch rule
- Lambda Layer
- IAM 권한

## 12. 핵심 설계 포인트

SAGO AI의 핵심 설계 포인트는 다음과 같습니다.

첫째, 사고 확률을 직접 예측하지 않고 유사 사고 사례를 검색합니다. 이 방식은 데이터가 사고 발생 건 중심으로 구성된 현재 상황에 적합하고, 결과를 사람이 이해하기 쉽습니다.

둘째, 고객 사고와 직원 사고를 분리합니다. 두 사고군은 원인과 대응 방식이 다르므로 별도 모델과 별도 가이드를 생성합니다.

셋째, LLM 입력에 단순 사례 목록만 넣지 않습니다. 오늘 조건, 발생 당시 조건, 기상 차이, 매장 환경, 리프 위험 해석을 함께 넣어 사고 원인과 재현 가능성을 판단하게 합니다.

넷째, LLM의 출력은 Tool Use 스키마로 강제합니다. 덕분에 프론트엔드와 카카오 메시지에서 안정적으로 사용할 수 있는 구조화된 결과를 얻습니다.

다섯째, 이미지 연결은 LLM이 아니라 코드가 처리합니다. LLM은 대표 사례를 고르고, 시스템은 `incident_id`를 기준으로 이미지 URL을 붙여 결과의 일관성을 유지합니다.

여섯째, 런타임은 JSON 모델만 사용합니다. 학습 단계에서는 scikit-learn을 쓰지만 Lambda에서는 순수 Python 트리 실행기를 사용해 배포 의존성을 줄였습니다.
