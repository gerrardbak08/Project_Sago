# 다이소 매장 안전사고 예방 AI 시스템 — 서비스 요구사항 v3

## 1. 비즈니스 목표

- 기상 + 과거 사고 데이터 분석 기반 **선제적 안전 알림** 시스템
- 전년도 112건 산업재해 **10% 이상 절감**, 중대재해 **0건** 달성
- **일 단위 배치**: 매일 아침 기상 예보 + 매장 정보 기반 알림 발송

---

## 2. 시스템 아키텍처

```
[오프라인 — 1회 구축 + 주기적 갱신]

  사고 데이터 (CUST 1,471건 + EMP 447건)
       ↓
  데이터 정제 + 피처 엔지니어링
       ↓
  Decision Tree 학습 (CUST/EMP 각각)
    - X: 기상 + 매장 피처
    - y: 사고유형 (분기 기준으로만 사용, 예측 목적 아님)
    - min_samples_leaf=10
       ↓
  리프 노드별 사고 사례 테이블 구축 (JSON)
    - 각 리프: 조건 규칙 + 사고 통계 + 전체 사고 사례 리스트

[온라인 — 매일 아침 배치]

  기상청 API (오늘 예보) + 매장 메타 DB
       ↓
  피처 전처리 (동일 파이프라인)
       ↓
  tree.apply() → 리프 노드 ID → 해당 리프의 사고 사례 조회
       ↓
  Fallback (리프 사례 부족 시 부모 노드로 롤업)
       ↓
  Bedrock LLM: 사고 사례 + 기상 + 매장 정보 → 안전 가이드 생성
       ↓
  카카오 알림톡 발송
```

## 3. 데이터 설계

### 3.1 핵심 접근: 사고 건만 사용

사고 미발생일을 포함한 전체 데이터(110만 행)가 아닌, **사고가 발생한 건만(1,918행)** 사용한다. "사고가 나는가?"를 예측하는 것이 아니라 "이 조건에서 과거 어떤 사고가 있었는가?"를 조회하는 구조이므로, 사고 미발생 행은 불필요하다.

| 데이터 | 설명 | 행 수 |
|--------|------|-------|
| `store_cust_incident_weather.csv` | 직원사고(고객CS) — 사고 건만 | 1,471건 |
| `store_emp_incident_weather.csv` | 고객사고(산업재해) — 사고 건만 | 447건 |

### 3.2 Decision Tree 피처 (X)

서빙 시점에 가용한 데이터만 피처로 사용한다.

**기상 피처 (Tier 1 — 필수)**:
- 최저기온(°C), 최고기온(°C), 평균 상대습도(%)
- 일강수량(mm), 일 최심적설(cm)

**기상 피처 (Tier 2 — 보조)**:
- 강수 계속시간(hr), 1시간 최다강수량(mm)
- 최대 순간 풍속(m/s), 평균 지면온도(°C), 최저 초상온도(°C)

**매장 피처**:
- 평수, 실평수, 진열평수, 창고, 계약면적(㎡)
- 형태 (직영점/유통점/유통행사)
- 구분 (단품관리/금액관리)
- 지역 (28개 팀)

**결측치 처리**:
- 강수/적설 NaN → 0 (비/눈 없음)
- 기타 기상 결측 → ffill/bfill

### 3.3 Decision Tree 라벨 (y) — 분기 기준 전용

사고유형을 라벨로 사용하되, **예측 정확도가 목적이 아니라 사고유형 분포 차이가 극대화되는 분기점을 자동 탐색하는 도구**로 Decision Tree를 활용한다.

- **CUST (직원→고객CS)**: `사고유형` (5종: 낙상, 재물, 충돌, 자상, 클레임)
- **EMP (고객→산업재해)**: `재해 유형` (13종 → 주요 7종: 넘어짐, 무리한 동작, 물체에 맞음, 베임, 부딪힘, 떨어짐, 기타)

### 3.4 리프 노드에 저장되는 사고 사례 데이터

각 리프 노드에 해당 조건의 사고 사례를 **전부** 저장한다.

**CUST 사고 사례 컬럼**:
- 발생일시, 사고내용요약, 사고유형, 장소, 원인1, 원인2, 원인3, 비고
- 기상 데이터 (Tier 1 + Tier 2 전체)
- 매장 정보 (매장, 지역, 형태, 구분, 평수 등)

**EMP 사고 사례 컬럼**:
- 발생일시, 재해 종류, 재해 유형, 기인물, 상병명, 사고 내용
- 기상 데이터 (Tier 1 + Tier 2 전체)
- 매장 정보 (매장, 지역, 형태, 구분, 평수 등)

### 3.5 데이터 누출 방지

사후 정보 컬럼(처리과정, 처리결과, 보상금액, 종결일시 등)은 피처(X)에 사용하지 않으며, 사고 사례 JSON에서도 제외한다.

## 4. Decision Tree 기반 조건부 사고 사례 테이블

### 4.1 선택 근거

| 요구사항 | Decision Tree 적합성 |
|----------|---------------------|
| 사고유형 분포 차이가 큰 조건 분기 | Gini/Entropy 기준으로 자동 탐색 |
| 연속형 + 범주형 혼합 | 네이티브 지원 (분할점 자동 결정) |
| 리프 노드 = 사고 사례 그룹 | `tree.apply()`로 즉시 조회 |
| 분기 규칙의 해석 가능성 | "최저기온 < 0 & 일강수량 > 0" 같은 규칙 추출 가능 |
| 최소 사례 수 보장 | `min_samples_leaf` 파라미터로 제어 |
| 외부 의존성 없음 | scikit-learn만으로 구현 |

### 4.2 하이퍼파라미터

```python
DecisionTreeClassifier(
    min_samples_leaf=10,    # 리프 노드 최소 10건 보장
    criterion='gini',       # 사고유형 분포 차이 기준
    random_state=42,
)
```

- `min_samples_leaf=10`: 각 리프에 최소 10건의 사고 사례 보장. 이보다 적으면 분기 중지.
- `max_depth`, `max_leaf_nodes`: 제한 없음 (min_samples_leaf가 자연스럽게 트리 크기 제어)
- CUST/EMP 각각 별도 트리 학습

### 4.3 학습 프로세스

```python
# 1. 데이터 로드 + 전처리
df, X, feature_names = preprocess_pipeline(source)

# 2. 라벨 (분기 기준 전용)
y = df[type_col]  # CUST: '사고유형', EMP: '재해 유형'

# 3. Decision Tree 학습
tree = DecisionTreeClassifier(min_samples_leaf=10, random_state=42)
tree.fit(X, y)

# 4. 각 학습 데이터의 리프 ID 저장
leaf_ids = tree.apply(X)

# 5. 리프별 사고 사례 + 통계 → JSON 저장
```

## 5. 사고 사례 조회 시스템

### 5.1 조회 흐름

```
입력: 오늘 기상 예보 + 매장 메타
  ↓
전처리 (동일 파이프라인)
  ↓
tree.apply(X_today) → 리프 노드 ID
  ↓
해당 리프의 사고 사례 전체 조회
  ↓
사고유형 분포 + 대표 사례 → LLM 전달
```

### 5.2 Fallback: 부모 노드 롤업

리프 노드의 사례가 부족하거나 (min_samples_leaf=10이므로 최소 10건 보장), 더 넓은 맥락이 필요한 경우 트리의 부모 노드로 롤업한다.

```
Level 0: 리프 노드 (최소 10건)
  → 기본 조회. 대부분 여기서 해결.

Level 1: 부모 노드
  → 리프의 형제 노드 사례까지 포함
  → 트리 구조에서 한 단계 위로 올라감

Level 2: 글로벌 Fallback
  → CUST/EMP 전체 사고 데이터의 빈도 상위 유형 + 범용 안전 수칙
  → 알림에 "일반 안전 가이드"임을 명시
```

### 5.3 위험도 산출

리프 노드의 사고 건수와 사고유형 집중도를 기반으로 위험도를 산출한다.

```python
# 리프 노드 사고 건수 기반
leaf_count = len(leaf_incidents)
total_count = len(all_incidents)
frequency_score = leaf_count / total_count  # 해당 조건의 사고 빈도

# 사고유형 집중도 (상위 유형 비율이 높을수록 특정 위험이 명확)
dominant_ratio = top_type_count / leaf_count

# 종합 위험도
risk_score = frequency_score * weight_freq + dominant_ratio * weight_dom
```

알림 발송 기준:
- 모든 매장에 발송 (조건에 해당하는 사고 사례가 항상 존재하므로)
- 리프 노드의 사고 건수가 많을수록 상세 가이드, 적을수록 일반 가이드

### 6.2 Hallucination 방지

- 모든 수치/통계는 리프 노드에서 사전 계산된 값만 사용
- 유사 사고 텍스트를 "인용"하도록 프롬프트 설계 (LLM이 창작하지 않게)
- `rule_readable`을 전달하여 조건 매칭 근거를 가이드에 반영
- JSON 스키마 강제 + 파싱 에러 핸들링

### 6.3 LLM 모델 및 비용

- 모델: Amazon Bedrock Claude Sonnet 4
- 예상 호출: 일 200~300건 (전체 1,337 매장 중 알림 대상)
- Mock 모드: Bedrock 자격증명 없으면 자동 전환

## 7. 수용 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| AC-1 | 데이터 전처리 파이프라인이 사고 건을 정확히 로드하고 피처를 구성한다 | CUST 1,471건 / EMP 447건 확인, 피처 결측률 < 1% |
| AC-2 | Decision Tree 리프 노드 최소 사례 수 ≥ 10건 | `min(leaf_sizes) >= 10` |
| AC-3 | 리프 노드 내 사고유형 최빈 비율이 전체 비율보다 높다 (차별화) | 각 리프의 dominant type 비율 > 전체 dominant type 비율 |
| AC-4 | 리프 노드 수가 적정 범위 (CUST 20~80, EMP 10~40) | `tree.get_n_leaves()` |
| AC-5 | 모든 학습 데이터가 리프 노드에 정상 배정된다 | `len(tree.apply(X)) == len(X)` |
| AC-6 | 새로운 기상+매장 데이터 입력 시 리프 노드 조회가 정상 동작한다 | 임의 매장+날짜 조합 10건 테스트 |
| AC-7 | 생성된 안전 가이드 JSON이 필수 필드를 포함한다 | 샘플 20건 생성 → 스키마 검증 |
| AC-8 | 리프 노드의 사고유형과 가이드 내용이 일치한다 | 샘플 20건 → 리프 dominant type vs 가이드 언급 유형 비교 |
| AC-9 | E2E 파이프라인: 매장+날짜 → 알림 메시지 출력 완료 | API 호출 확인 |
| AC-10 | 신규 매장 입력 시 정상 동작 (트리는 매장코드가 아닌 매장 속성으로 분기) | 미존재 매장의 속성값으로 테스트 |

## 8. 리스크 및 완화 방안

| 리스크 | 심각도 | 완화 방안 |
|--------|--------|-----------|
| 1,918건으로 트리 과적합 | 중 | `min_samples_leaf=10`으로 제어. 리프 수 모니터링 |
| 기상 예보값 vs 실측값 괴리 | 중 | Decision Tree가 연속형 분할점을 자동 결정하므로 이산화 오차에 강건. 예보 오차 범위 내에서 같은 리프에 도달할 가능성 높음 |
| LLM Hallucination | 상 | 리프 노드의 실제 사고 사례를 "인용"하도록 프롬프트 설계. 수치는 사전 계산값만 주입 |
| 특정 리프에 사례 편중 | 중 | 트리 학습 후 리프 크기 분포 확인. 과대 리프는 `max_leaf_nodes` 조정으로 추가 분할 유도 |
| 사고 데이터 추가 시 재학습 필요 | 하 | 오프라인 파이프라인 자동화. 분기 1회 재구축 권장. Decision Tree 학습은 수 초 내 완료 |
| EMP 447건으로 트리 깊이 제한 | 중 | EMP는 리프 수가 적을 수 있음. 리프당 사례가 충분한지 확인 후 `min_samples_leaf` 조정 |
| 범주형 피처 처리 | 하 | sklearn DecisionTree는 범주형을 직접 지원하지 않으므로 LabelEncoder 또는 OrdinalEncoder 사용. 지역(28종)은 원핫 인코딩 또는 빈도 인코딩 검토 |
